Reorder Data in Log Files

Medium
GraphArrayStringSorting

You are given an array of logs. Each log is a space-delimited string of words, where the first word is the identifier.

There are two types of logs:

Letter-logs: All words (except the identifier) consist of lowercase English letters. Digit-logs: All words (except the identifier) consist of digits. Reorder these logs so that:

The letter-logs come before all digit-logs. The letter-logs are sorted lexicographically by their contents. If their contents are the same, then sort them lexicographically by their identifiers. The digit-logs maintain their relative ordering. Return the final order of the logs.

Example 1:

Input: logs = ["dig1 8 1 5 1","let1 art can","dig2 3 6","let2 own kit dig","let3 art zero"] Output: ["let1 art can","let3 art zero","let2 own kit dig","dig1 8 1 5 1","dig2 3 6"] Explanation: The letter-log contents are all different, so their ordering is "art can", "art zero", "own kit dig". The digit-logs have a relative order of "dig1 8 1 5 1", "dig2 3 6". Example 2:

Input: logs = ["a1 9 2 3 1","g1 act car","zo4 4 7","ab1 off key dog","a8 act zoo"] Output: ["g1 act car","a8 act zoo","ab1 off key dog","a1 9 2 3 1","zo4 4 7"]

Constraints:

1 <= logs.length <= 100 3 <= logs[i].length <= 100 All the tokens of logs[i] are separated by a single space. logs[i] is guaranteed to have an identifier and at least one word after the identifier.

Problem idea (short)

Each log has an identifier followed by content. Logs are either:

  • Letter-log: content words are letters, e.g. "let1 art can".
  • Digit-log: content words are digits, e.g. "dig1 8 1 5 1".

We must reorder logs so that:

  1. All letter-logs come before digit-logs.
  2. Letter-logs are sorted lexicographically by content, and if equal, by identifier.
  3. Digit-logs retain their original order.

The easiest and clearest approach is:

  1. Partition the input into letterLogs and digitLogs (keeping digitLogs in original order).
  2. Sort letterLogs with a comparator that compares by content first, then identifier.
  3. Concatenate sorted letterLogs followed by digitLogs.

This is stable and straightforward.

---

Why it‚Äôs correct

  • Partitioning ensures digit-logs are after all letter-logs.
  • Sorting letter-logs by (content, id) matches the required ordering.
  • Because we never reorder digit-logs, their relative order is preserved.

---

Complexity

Let n = logs.length and let L be the average length of a log string.

  • Partitioning: O(n * L) to scan each log and split identifier/content.
  • Sorting letter-logs: Suppose k letter-logs. Sorting cost O(k log k) comparisons, each comparison takes O(L) time to compare content/identifier in the worst case ‚Üí O(k log k * L).
  • Concatenation: O(n).

Overall: O(n L + k log k L) ‚âà O(n log n * L) in worst case, which is fine for constraints (n ‚â§ 100).

Space: O(n) extra for the partitions.

---

Implementation choices

  1. Split-then-sort (recommended): parse each log (one indexOf(' ') to split identifier/content), store letter logs as objects (content, id, fullLog), collect digit logs as strings. Sort letter objects by (content, id) and then build answer.
  2. One-pass sort with custom comparator: sort the original array with a comparator that detects type and compares accordingly. This can be done but may repeatedly parse the same log strings during comparisons (which is fine for small n but a little less efficient). Because Java's Arrays.sort is stable (TimSort), this is also acceptable; still I prefer the explicit partition for readability.

Java implementation (clean and tested)

import java.util.*;

/**
 * Reorder logs so that letter-logs come before digit-logs.
 * Letter-logs are ordered by content, then identifier.
 * Digit-logs remain in original order.
 */
class Solution {
    static class Log {
        String id;
        String content;
        String full;
        Log(String id, String content, String full) {
            this.id = id;
            this.content = content;
            this.full = full;
        }
    }

    public String[] reorderLogFiles(String[] logs) {
        List letterLogs = new ArrayList<>();
        List digitLogs = new ArrayList<>();

        for (String log : logs) {
            int firstSpace = log.indexOf(' ');
            String id = log.substring(0, firstSpace);
            String rest = log.substring(firstSpace + 1);
            // Determine whether this is a digit-log or letter-log by checking first char of content
            if (Character.isDigit(rest.charAt(0))) {
                digitLogs.add(log); // keep original order
            } else {
                letterLogs.add(new Log(id, rest, log));
            }
        }

        // Sort letter-logs by content then by id
        Collections.sort(letterLogs, new Comparator() {
            @Override
            public int compare(Log a, Log b) {
                int cmp = a.content.compareTo(b.content);
                if (cmp != 0) return cmp;
                return a.id.compareTo(b.id);
            }
        });

        // Build the result: sorted letter logs first, then digit logs in original order
        String[] ans = new String[logs.length];
        int idx = 0;
        for (Log lg : letterLogs) ans[idx++] = lg.full;
        for (String dg : digitLogs) ans[idx++] = dg;
        return ans;
    }

    // quick test
    public static void main(String[] args) {
        Solution sol = new Solution();
        String[] logs1 = {"dig1 8 1 5 1","let1 art can","dig2 3 6","let2 own kit dig","let3 art zero"};
        System.out.println(Arrays.toString(sol.reorderLogFiles(logs1)));
        // expected: ["let1 art can","let3 art zero","let2 own kit dig","dig1 8 1 5 1","dig2 3 6"]

        String[] logs2 = {"a1 9 2 3 1","g1 act car","zo4 4 7","ab1 off key dog","a8 act zoo"};
        System.out.println(Arrays.toString(sol.reorderLogFiles(logs2)));
        // expected: ["g1 act car","a8 act zoo","ab1 off key dog","a1 9 2 3 1","zo4 4 7"]
    }
}

Variations / small optimizations

  • If you want slightly fewer allocations, you can store index positions and do substring only once per log (the code above already does that).
  • For large inputs and repeated comparisons, consider caching parsed components; but not needed here.
