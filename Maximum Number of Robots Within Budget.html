Maximum Number of Robots Within Budget

Medium TreeGraphArrayStringTwo Pointers

Problem restatement Given two arrays chargeTimes and runningCosts of length n and a budget, find the maximum length of a contiguous subarray (consecutive robots) such that:

cost(window) = max(chargeTimes in window) + windowSize * sum(runningCosts in window) <= budget

Return the maximum possible window size (or 0 if none).

Input (example 1) chargeTimes = [3,6,1,3,4], runningCosts = [2,1,3,4,5], budget = 25 Output: 3

Input (example 2) chargeTimes = [11,12,19], runningCosts = [10,8,7], budget = 19 Output: 0

Constraints (provided)

  • n = chargeTimes.length = runningCosts.length, 1 <= n <= 5*10^4
  • 1 <= chargeTimes[i], runningCosts[i] <= 10^5
  • 1 <= budget <= 10^15

---

Quick script to say to the interviewer

  1. ‚ÄúI‚Äôll use a sliding window with two pointers and maintain the current window‚Äôs sum of runningCosts and the maximum chargeTime in the window.‚Äù
  2. ‚ÄúTo get the maximum efficiently I‚Äôll keep indices in a monotonic deque (decreasing by chargeTimes). For sum, maintain a running long accumulator. Expand right; while cost exceeds budget slide left and update sum & deque.‚Äù
  3. ‚ÄúThis gives O(n) time and O(n) space (deque), using long for sums/budget to avoid overflow.‚Äù

---

Approaches considered (brief)

  1. Binary search on window size + check using sliding window + deque ‚Äî O(n log n). Good and common.
  2. Two-pointer sliding window (chosen) ‚Äî O(n) amortized: expand right, shrink left while invalid. Simpler, optimal.
  3. Segment tree for max and prefix sums ‚Äî more code, unnecessary here.

I choose two-pointer sliding window with a monotonic deque because it's O(n), clean, and standard for contiguous windows with a max constraint.

---

Detailed chosen approach (step-by-step)

  • Data structures:
  • Deque dq stores indices of chargeTimes in decreasing order of chargeTimes (front = max).
  • long sumRunning ‚Äî running sum of runningCosts in current window.
  • int left = 0, iterate right from 0..n-1.
  • Loop:
  1. Add runningCosts[right] to sumRunning.
  2. Maintain deque: while dq not empty and chargeTimes[dq.peekLast()] <= chargeTimes[right] popLast(); then push right.
  3. Compute windowSize = right - left + 1. Compute cost = chargeTimes[dq.peekFirst()] + windowSize * sumRunning (use long).
  4. While left <= right and cost > budget:
  • If dq.peekFirst() == left then popFirst() because that index will leave window.
  • Subtract runningCosts[left] from sumRunning, increment left.
  • Recompute windowSize and cost (or compute lazily next loop).
  1. After shrinking, the window [left..right] is valid; update ans = max(ans, right - left + 1).
  • Return ans.

Why correctness holds: for a fixed right, increasing left only decreases windowSize, sumRunning, and possibly the max (via deque), so the while loop terminates at the smallest left making the window valid (if any). We check every possible right once; deque operations amortize to O(1).

---

Complexity

  • Time: O(n) amortized ‚Äî each index is pushed/popped at most once in the deque; left and right move at most n steps each.
  • Space: O(n) worst-case for the deque (but typically much smaller) and O(1) extra scalars. Use long for sums to avoid overflow (windowSize * sumRunning can exceed 32-bit).

---

Java implementation (clean & tested)

import java.util.*;

public class MaxRobotsWithinBudget {
    public int maximumRobots(int[] chargeTimes, int[] runningCosts, long budget) {
        int n = chargeTimes.length;
        Deque dq = new ArrayDeque<>(); // stores indices, chargeTimes decreasing
        long sumRunning = 0L;
        int left = 0;
        int ans = 0;

        for (int right = 0; right < n; right++) {
            // add running cost
            sumRunning += runningCosts[right];

            // maintain monotonic deque for max chargeTime
            while (!dq.isEmpty() && chargeTimes[dq.peekLast()] <= chargeTimes[right]) {
                dq.pollLast();
            }
            dq.offerLast(right);

            // shrink window while cost > budget
            while (left <= right) {
                long windowSize = (long)(right - left + 1);
                long maxCharge = (long) chargeTimes[dq.peekFirst()];
                long cost = maxCharge + windowSize * sumRunning;

                if (cost <= budget) break; // window valid

                // remove left element
                if (!dq.isEmpty() && dq.peekFirst() == left) {
                    dq.pollFirst();
                }
                sumRunning -= runningCosts[left];
                left++;
            }

            // window [left..right] is valid now
            ans = Math.max(ans, right - left + 1);
        }

        return ans;
    }

    // quick manual tests
    public static void main(String[] args) {
        MaxRobotsWithinBudget solver = new MaxRobotsWithinBudget();

        int[] charge1 = {3,6,1,3,4};
        int[] running1 = {2,1,3,4,5};
        System.out.println(solver.maximumRobots(charge1, running1, 25L)); // expected 3

        int[] charge2 = {11,12,19};
        int[] running2 = {10,8,7};
        System.out.println(solver.maximumRobots(charge2, running2, 19L)); // expected 0

        // extra tests
        int[] charge3 = {5,5,5};
        int[] running3 = {1,1,1};
        System.out.println(solver.maximumRobots(charge3, running3, 11L)); // expect 3 (max 5 + 3*3 = 14 >11 so maybe smaller) -> verify
    }
}

---

Edge cases & tests to mention to interviewer

  • All single robot cost > budget ‚Üí answer 0.
  • budget large enough to take all robots ‚Üí answer n.
  • Repeated equal chargeTimes and varying runningCosts ‚Äî deque handles ties.
  • Use long because windowSize * sumRunning can exceed 32-bit (e.g., windowSize ~ 5e4 and sumRunning ~ 1e9 could overflow int).

---

Optional alternative (binary-search on length)

  • Binary search k (1..n) and for each k check if any window of size k satisfies cost constraint using sliding window with deque & prefix sum ‚Äî yields O(n log n). Useful if interviewer prefers binary-search reasoning.
