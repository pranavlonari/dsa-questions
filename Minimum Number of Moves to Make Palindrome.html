Minimum Number of Moves to Make Palindrome

You are given a string s consisting only of lowercase English letters.

In one move, you can select any two adjacent characters of s and swap them.

Return the minimum number of moves needed to make s a palindrome.

Note that the input will be generated such that s can always be converted to a palindrome.

Example 1:

Input: s = "aabb" Output: 2 Explanation: We can obtain two palindromes from s, "abba" and "baab".

  • We can obtain "abba" from s in 2 moves: "aabb" -> "abab" -> "abba".
  • We can obtain "baab" from s in 2 moves: "aabb" -> "abab" -> "baab".
Thus, the minimum number of moves needed to make s a palindrome is 2. Example 2:

Input: s = "letelt" Output: 2 Explanation: One of the palindromes we can obtain from s in 2 moves is "lettel". One of the ways we can obtain it is "letelt" -> "letetl" -> "lettel". Other palindromes such as "tleelt" can also be obtained in 2 moves. It can be shown that it is not possible to obtain a palindrome in less than 2 moves.

Constraints:

1 <= s.length <= 2000 s consists only of lowercase English letters. s can be converted to a palindrome using a finite number of moves.

Nice ‚Äî this is a classic problem. Below is an interview-ready answer you can say to the interviewer, followed by a clear, correct Java implementation (greedy simulation), complexity, tests, and a short note about a faster alternative.

---

Problem restatement

Given a string s of lowercase letters, in one move you may swap any two adjacent characters. Return the minimum number of moves required to make s a palindrome. Input guarantees it is possible.

Input example: s = "aabb" Output example: 2

Constraints: 1 <= s.length <= 2000, s lowercase, convertible to palindrome.

---

Short script to say to the interviewer

‚ÄúI‚Äôll use a two-pointer greedy simulation. Keep l at start and r at end. If s[l] == s[r] move inward. Otherwise search from r down to l for a match to s[l]. If found at k, bubble that character rightwards to r using r-k adjacent swaps (count them). If no match (i.e., k == l), the char at l must be the unique middle character ‚Äî swap it one step right (l with l+1) and count 1 move, then retry (this moves the unique char toward center). Repeat until l >= r. This greedy produces the minimum number of adjacent swaps.‚Äù

(If asked: mention there is also an O(n log n) solution mapping characters to target indices and counting inversions with a BIT ‚Äî useful for large input, but greedy is simpler and runs fine for n ‚â§ 2000.)

---

Why the greedy works (intuition)

Each time we match a left character with the rightmost available same character, moving that match to the rightmost position uses the minimum number of adjacent swaps to place that particular letter at the needed spot without reordering previously fixed outer letters. If the left character has no partner to the right it must be the unique middle character ‚Äî moving it one step right is optimal to eventually place it in middle.

---

Complexity

  • Time: O(n¬≤) worst-case ‚Äî for each left position you may scan to find its match and perform up to O(n) swaps. For n ‚â§ 2000 this is fine.
  • Space: O(n) due to the character array (in-place simulation).

---

Java implementation (greedy simulation)

import java.util.*;

public class MinMovesToMakePalindrome {
    public int minMovesToMakePalindrome(String s) {
        int n = s.length();
        char[] arr = s.toCharArray();
        int l = 0, r = n - 1;
        int moves = 0;

        while (l < r) {
            if (arr[l] == arr[r]) {
                l++;
                r--;
                continue;
            }

            // find match for arr[l] from the right side
            int k = r;
            while (k > l && arr[k] != arr[l]) k--;

            if (k == l) {
                // no match on the right: arr[l] is the unique middle char
                // swap it one step to the right (with arr[l+1]) and count 1 move
                swap(arr, l, l + 1);
                moves++;
                // don't move l or r: we need to re-evaluate arr[l] (new char) in next iteration
            } else {
                // bring arr[k] to position r by swapping adjacent pairs
                while (k < r) {
                    swap(arr, k, k + 1);
                    moves++;
                    k++;
                }
                // now arr[l] and arr[r] match; move inward
                l++;
                r--;
            }
        }

        return moves;
    }

    private void swap(char[] arr, int i, int j) {
        char tmp = arr[i];
        arr[i] = arr[j];
        arr[j] = tmp;
    }

    // quick tests
    public static void main(String[] args) {
        MinMovesToMakePalindrome solver = new MinMovesToMakePalindrome();

        System.out.println(solver.minMovesToMakePalindrome("aabb"));    // 2
        System.out.println(solver.minMovesToMakePalindrome("letelt"));  // 2
        System.out.println(solver.minMovesToMakePalindrome("mbadm"));   // 2 (e.g., "madbm" -> "madbm"? depends; valid test)
        System.out.println(solver.minMovesToMakePalindrome("a"));       // 0
        System.out.println(solver.minMovesToMakePalindrome("racecar")); // 0 (already palindrome)
    }
}

---

Notes & alternatives

  • The greedy simulation is simple to implement and intuitive. It is accepted for the constraints here.
  • If you need better asymptotic performance for much larger n, you can:
  1. Decide the target palindrome arrangement (one of many), map each character occurrence in the original string to a target index (build arrays of indices for each character), and then compute how many adjacent swaps are needed to reorder into that target ‚Äî which equals the number of inversions between the original positions and the target positions.
  2. Count inversions using a Fenwick Tree (BIT) or segment tree in O(n log n). That approach is more complex but gives O(n log n) runtime.
