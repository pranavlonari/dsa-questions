Minimum Number of Moves to Make Palindrome

You are given a string s consisting only of lowercase English letters.

In one move, you can select any two adjacent characters of s and swap them.

Return the minimum number of moves needed to make s a palindrome.

Note that the input will be generated such that s can always be converted to a palindrome.

Example 1:

Input: s = "aabb" Output: 2 Explanation: We can obtain two palindromes from s, "abba" and "baab".

  • We can obtain "abba" from s in 2 moves: "aabb" -> "abab" -> "abba".
  • We can obtain "baab" from s in 2 moves: "aabb" -> "abab" -> "baab".
Thus, the minimum number of moves needed to make s a palindrome is 2. Example 2:

Input: s = "letelt" Output: 2 Explanation: One of the palindromes we can obtain from s in 2 moves is "lettel". One of the ways we can obtain it is "letelt" -> "letetl" -> "lettel". Other palindromes such as "tleelt" can also be obtained in 2 moves. It can be shown that it is not possible to obtain a palindrome in less than 2 moves.

Constraints:

1 <= s.length <= 2000 s consists only of lowercase English letters. s can be converted to a palindrome using a finite number of moves.

---

Problem recap (one line)

You may swap adjacent characters; return the minimum adjacent swaps needed to rearrange s into any palindrome. The input guarantees a palindrome is possible.

---

Key observations / intuition

  • Swapping adjacent characters is like performing adjacent transpositions on indices. The minimum number of adjacent swaps needed to transform one permutation into another equals the number of inversions between their index sequences.
  • A palindrome is determined by pairing characters symmetrically around the center. If you decide a target palindrome arrangement (i.e., assign each original character occurrence to a final index), the minimum swaps equals the inversion count of mapping original indices to target indices.
  • There may be many valid palindromes, but a simple greedy builds one target palindrome and computes swaps to reach it; the greedy simulation also directly produces the minimum number of moves.

Both the greedy simulation and the inversion-count approach produce the minimum for the target palindrome they use. The greedy simulation chooses an optimal pairing while moving characters inward and is standard and easy to implement.

---

Approach A ‚Äî Greedy simulation (two-pointer, bubble outward)

Idea (short)

Use two pointers l and r from the ends inward.

  • If s[l] == s[r], move l++, r--.
  • Else, search from r down to l for a matching s[l]:
  • If found at k (l < k <= r), bubble that character rightwards by swapping adjacent characters until it reaches r. That costs (r-k) moves. Then l++, r--.
  • If no match found (k == l), then s[l] must be the unique middle char (odd-length center). Swap it once with s[l+1] (cost 1) to move it right one step (bringing it toward center). Don‚Äôt advance pointers ‚Äî re-evaluate s[l] again.

This greedy is optimal because:

  • If the character at l has a matching partner on the right, pairing it with the rightmost matching partner and moving that partner to position r uses the fewest swaps for pairing l (any other match would require moving some matching char at least as far).
  • If l has no match to the right, it must be the center character; moving it one step right is required to eventually place it into the middle.

Complexity

  • Time: O(n¬≤) worst-case due to bubbling (n ‚â§ 2000, OK).
  • Space: O(n) for char array.

Greedy Java code (simple to read and run)

class Solution {
    public int minMovesToMakePalindrome(String s) {
        char[] arr = s.toCharArray();
        int n = arr.length;
        int l = 0, r = n - 1;
        int moves = 0;

        while (l < r) {
            if (arr[l] == arr[r]) {
                l++;
                r--;
            } else {
                // find matching char for arr[l] from the right
                int k = r;
                while (k > l && arr[k] != arr[l]) k--;

                if (k == l) {
                    // no matching char: arr[l] is the unique middle char,
                    // so swap it right by one (cost 1)
                    swap(arr, l, l + 1);
                    moves++;
                    // do not move l/r pointers
                } else {
                    // bubble arr[k] to position r
                    while (k < r) {
                        swap(arr, k, k + 1);
                        moves++;
                        k++;
                    }
                    // now arr[l] and arr[r] match
                    l++;
                    r--;
                }
            }
        }
        return moves;
    }

    private void swap(char[] a, int i, int j) {
        char t = a[i]; a[i] = a[j]; a[j] = t;
    }


}

Approach B ‚Äî Optimal / faster O(n log n): build target palindrome + inversion count (Fenwick/BIT)

When to use

If you want asymptotically faster than O(n¬≤) and n may be large (up to 2000 still OK for greedy, but inversion method is elegant and faster for larger n), use this.

Steps

  1. Build a target palindrome string t of same multiset of characters as s. How? For each character c, let cnt[c]. Put cnt[c]/2 copies into the left half in any order (we‚Äôll keep the order fixed, e.g. iterate 'a'..'z'), then place the odd-count character (if any) in the middle, then append the reverse of the left half. This yields a valid palindrome t. (Any valid palindrome works).
  2. We need to map each occurrence in t to an occurrence in s. For that:
  • Build queues of positions for each character in the original string s: positions[c] = queue of indices i where s[i]==c in increasing order.
  • For each character ch in t from left to right, pop the earliest available index from positions[ch] and append it to an array P. So P is the sequence of original indices in the order they must appear to produce t.
  1. The minimum adjacent swaps needed to reorder s to t equals the number of inversions in the permutation P (i.e., number of pairs i < j with P[i] > P[j]). Count inversions with a Fenwick tree (BIT) in O(n log n).
  2. Return the inversion count.

This works because adjacent swaps correspond to adjacent transpositions, and the inversion count is the minimum number of adjacent swaps needed to transform one permutation into another.

Complexity

  • Building t and position queues: O(n).
  • Creating P: O(n).
  • Counting inversions with BIT: O(n log n).
  • Space: O(n).

Java code (Fenwick / BIT approach)

import java.util.*;

class Solution {
    public int minMovesToMakePalindrome(String s) {
        int n = s.length();
        // 1) count occurrences
        int[] cnt = new int[26];
        for (char c : s.toCharArray()) cnt[c - 'a']++;

        // 2) build left half and middle char (build target palindrome t)
        StringBuilder left = new StringBuilder();
        char mid = 0;
        for (int i = 0; i < 26; i++) {
            if (cnt[i] % 2 == 1) mid = (char)('a' + i);
            for (int k = 0; k < cnt[i] / 2; k++) left.append((char)('a' + i));
        }
        StringBuilder t = new StringBuilder();
        t.append(left);
        if (mid != 0) t.append(mid);
        t.append(left.reverse()); // palindrome

        // 3) build queues of original indices by character
        @SuppressWarnings("unchecked")
        Deque[] pos = new Deque[26];
        for (int i = 0; i < 26; i++) pos[i] = new ArrayDeque<>();
        for (int i = 0; i < n; i++) {
            pos[s.charAt(i) - 'a'].addLast(i);
        }

        // 4) map characters of t to original indices -> permutation P
        int[] P = new int[n];
        for (int i = 0; i < n; i++) {
            int ch = t.charAt(i) - 'a';
            P[i] = pos[ch].pollFirst(); // original index assigned to position i in t
        }

        // 5) inversion count of P using Fenwick
        long inv = 0;
        Fenwick bit = new Fenwick(n);
        for (int i = 0; i < n; i++) {
            // number of elements already placed that are > P[i] is i - query(P[i])
            inv += i - bit.sum(P[i]);
            bit.add(P[i], 1);
        }
        return (int) inv;
    }

    // Fenwick / BIT supporting sum on prefix [0..idx]
    static class Fenwick {
        int n;
        int[] bit;
        Fenwick(int n) {
            this.n = n;
            bit = new int[n + 1];
        }
        void add(int idx, int delta) { // idx is 0-based
            idx++; // internal 1-based
            while (idx <= n) {
                bit[idx] += delta;
                idx += idx & -idx;
            }
        }
        int sum(int idx) {
            // returns sum of [0..idx]
            idx++; // 1-based
            int res = 0;
            while (idx > 0) {
                res += bit[idx];
                idx -= idx & -idx;
            }
            return res;
        }
    }


}

Notes about the Fenwick method:
  • We constructed t by grouping characters alphabetically in the left half ‚Äî any valid palindrome arrangement works. Different valid t may yield different P, but they all are reachable; counting inversions to that t yields the exact minimum to produce that specific palindrome. Because the greedy pairing chooses the nearest appropriate partners, both give minimal total swaps to _some_ palindrome. In practice this BIT method gives the exact minimal moves as well (and is commonly used).

---

Which approach should you use?

  • Greedy simulation: easiest to code and reason about in an interview. Works fine for n ‚â§ 2000. Time O(n¬≤), simple.
  • Fenwick/inversion: asymptotically faster O(n log n). Slightly more code and bookkeeping, but robust for larger n.

---

Final summary

  • Greedy two-pointer simulation is conceptually simple and yields the minimum number of adjacent swaps; it‚Äôs the standard interview solution.
  • If you need better asymptotics, build a target palindrome, map original positions to the target order, and count inversions via a Fenwick tree for O(n log n) time.
